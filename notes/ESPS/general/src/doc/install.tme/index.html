    ETMS8614,  Entropic  Signal Processing System (ESPS) Pro
gramming Guidelines

1.  Introduction

     This document describes the installation procedure  for
ESPS Version 3.  Please read this document completely before
attempting the installation.    This release is intended for
use  on  Masscomp  5000 or 6000 series computers running RTU
3.0 or greater, Sun3, Sun4 or Sun386i  workstations  run
ning Sun OS 3.2 or greater, HP 800 and 300 series computers,
DEC  DecStation  3100 (called the DS3100 here) workstations,
and Convex computers.    The  installation  script  requires
that a number of standard UNIX utilities be available and on
the  standard  execution path, such as grep, cc, nroff, sed,
make, diffmk and test.

     Earlier versions of ESPS required Fortran, but this re
lease does not.

     In the case of Sun386i some optional clusters  are  re
quired  to  build  ESPS from this source kit.  You must have
the development cluster installed (both the  base_devel  and
the sunview_devel part) and the doc_prep cluster.

2.  Overview of Installation Process

     The installation is simple.   The tape or diskettes are
read  into a clean directory (suggested name is esps).  This
directory will become the source directory and  can  by  lo
cated  anywhere on the file system, but it is suggested that
it be in the login directory of the local ESPS maintainer.

     The installer may adjust a few  installation  variables
in  the script to alter the default choices.   The installa
tion script creates a target directory and installs all ESPS
programs and related files there.   If your system is low on
file space, then the source directory can be removed.   How
ever, if you can afford the space, having the sources online
is  helpful, since they serve as example programs.  For con
venience, the target directory contains a symbolic link back
to the source directory.  No ESPS programs  require  any  of
the  files under the source directory after the installation
script has been properly run.

2.1.  ESPSs Place in the File System

     It is not necessary to be superuser  while  installing
ESPS.   It is not necessary (but it is possible) to put ESPS
files into the standard UNIX directories such  as  /usr/lib,
/usr/bin/,  and  /usr/include.  Whether you must have super
user access depends on where you want to install ESPS.   The
typical location for  the  target  directory  is  /usr/esps.













There  are two different ways to accomplish this.  One is to
have a superuser create the directory  /usr/esps  and  then
change the owner of it to the userid of the ESPS maintainer
(the  one  who  read in the tape and is going to run the in
stall script).  The better option is to have the  superuser
create  a  symbolic  link /usr/esps which points to the true
target directory location (for example  /home/esps),  create
this  directory  (make  sure  it is writtable by the userid
running the install script), and then set the target  direc
tory in the ESPS install script to /usr/esps.

     If  you have one of the waves+ products (other than for
the Sun386i) then you must have ESPS  rooted  at  /usr/esps.
This  will  be  requirement relaxed in a future release.  If
you do not have waves+ then /usr/esps is only  a  convention
and  you  can  install ESPS anywhere in your file system you
desire.   All file paths are determined by the  installation
script.

     On  a  Sun386i,  ESPS  with  waves+  must  be rooted at
/files/esps.  Note that it doesnt have to be  directly  in
stalled   there;   it  can  be  anywhere  and  connected  to
/files/esps with a symbolic link.

2.2.  A Word About Access Modes

     There are installation parameters  that  determine  the
access  modes of the files when they are installed.  The de
fault is write access by owner (should  be  the  local  ESPS
maintainer)  and  read  access  by others.  If you desire to
limit access to ESPS to a particular group,  you  can  reset
the  access  mode installation variables accordingly.  There
are installation parameters for the mode  of  the  programs,
manual pages, and library files.

2.3.  File Space Required

     This release of ESPS requires about 9 Megabytes of file
space  to  read  in  the distribution media.  This is 18,000
blocks on the Masscomp, HP, and DS3100, and 9,000 blocks  on
a  Sun.  The target directory requires about 17 Megabytes on
a Masscomp, about 26 Megabytes on a Sun3, HP or DS3100,  and
about  19 Megabytes on a Sun4 (Sparc) after all programs are
installed.  During installation, the source  directory  will
grow  by about 10 Megabytes owing to the .o files being cre
ated during compilation.  After the installation script  has
completed  correctly, you can remove thesource directory if
you are low on file space and do not need easy access to the
sources.

6.  Editing the Install Script

     There  are  several  variable  assignments that must be
checked by the installer.  They are the system type, the lo
cation of the target directory, and the data acquisition de
vice names.















     The variable MACH should be set to indicate the machine
type.  It is either M5500 for a 68010 Masscomp, M5600 for  a
68020  Masscomp, SUN3 for a Sun3, SUN4 for a Sun4, SUN386i
for a Sun386i, HP for an HP 300 or HP  800  series,  CONVEX
for  a  Convex  C1 or C2, or DS3100 for a DecStation 3100.
No other machine types are supported by this  release.   The
default value should be the same as the tape is labeled for,
but  if  you  ordered for several machine types, it wont be
right for all of them.  In any event, check the  MACH  vari
able in the install script.

     The  variable  SPSDIR is the location of the target di
rectory.   It is set to the full path of the desired  target
directory.    The  default  value (as shipped) is /usr/esps.
The install script will try to create this directory  if  it
does  not  exist.   Be sure that the userid running the in
stall script can write into the directory in which the  tar
get directory will be created.  If not (for example if it is
/usr)  have the superuser create it first and then chown of
it to the ESPS installers userid.

     The other installation variables that  might  need  ad
justment are the file access modes.  These default to access
by everyone on the system.  If you desire to limit access to
to  a  particular  group  you can set the access mode as de
sired.   The group used will be that of the installer.  (For
full access, the last digit of the mode should be  the  same
as  the next to last digit (eg. 0755 or 0644).  To limit ac
cess to the group, the last digit should be zero  (eg.  0750
or 0640).

     The  install script contains a listing at the beginning
of all variables that can be defined and their default  val
ues.   Note  that  the install script is a sh(1) script, and
there can be no spaces around the equal sign (=) in variable
assignment statements.

6.1.  Data Acquisition Devices on Masscomp (Concurrent)

     The variables DADEV, ADDEV, and CLKDEV are set  to  the
name of the digitaltoanalog converter, the analogtodigi
tal  converter,  and the clock device.   The d/a and a/d de
vice are the full path names of  the  device.   For  example
/dev/dacp0/daf0.   The  clock device is the full path of the
clock, excluding the trailing device  number.   For  example
/dev/dacp0/clk.   The default values are /dev/dacp0/daf0 for
DADEV, /dev/dacp0/adf0 for  ADDEV,  and  /dev/dacp0/clk  for
CLKDEV.

     MAXDA and MAXAD are set to the maximum (absolute) value
obtained  from your analogtodigital equipment, or that can
be sent to your digitaltoanalog  equipment.   The  default
value  is for a 12 bit twoscompliment device (2047).   This













value affects scaling that is done within the play  program.
It is not used by any of the signal processing programs.

6.2.  Specifying Window Systems

     If  you  are  on  a Masscomp system, you have three op
tions: SP40 Graphics, X Windows,  or  both.   All  Masscomp
systems  with  graphics terminals have at least one of these
window systems.  Determine which you have and set the  vari
ables  XWIN  and  SP40 to yes or no depending on whether you
have these.  If you do not have Masscomp graphics terminals,
we suggest that you set SP40 to YES and XWIN to NO.

     For HP and DS3100 systems, we assume that X windows  is
present.

     For  Sun systems, this version of the kit only supports
building ESPS for the Sunview window system.  A patch kit is
available to convert the installed ESPS (or a binary distri
bution) to X windows.   If it was not on  your  distribution
media and you need it, contact Entropic.

8.  Run the Install Script

     After you have checked and reset any installation vari
ables you simply run the script.   It is a good idea to cap
ture the standard error and standard output from the install
script  into a log file.  The following example assumes that
you are using the cshell( csh).
% cd
% cd esps/general
% install  >& log &

     To watch the log file during while the  install  script
runs do:
% tail f log

     This will run the install script and save all output in
the  file  named  log.   You can monitor the progress of the
install by using tail f.  (Tail shows the last few lines of
a file.  The f option causes it to print  additional  lines
as  they  are added to the file.  Note that tail with the f
option does not ever exit.  It has no way  of  knowing  that













the  file  being  watched  will not have any additional text
added to it.   If you are watching the  install  with  tail,
you  must  kill it with controlC, or whatever your UNIX in
terrupt character is, when you see  the  message  indicating
the  end  of  the  ESPS  install script.)   When the install
script completes it will print  the  message:  ESPS  general
part of installation done (followed by the data/time).

     If  something  goes  wrong and you need to stop the in
stallation process, just use the UNIX kill command  to  kill
the  process.   You can supply the process number(s) printed
after the install command, or the csh job number supplied by
the job command.
% jobs
 [1]  +Running   install  >& log
% kill %1

9.  Checking the Results

     The install script produces a large  amount  of  output
(the  log file will be about 100K bytes).   Many of the mes
sages from make might look like errors, but they are normal.
Each error message from make in the install log,  should  be
followed  by  a message from make saying that it is ignoring
the error.   All of these are expected.   Any unexpected er
rors will not be ignored and will cause the make to stop.  A
few C programs produce  warnings  about  unreachable  state
ments,  depending on some installation variables.  These can
be ignored.

     There might be a few messages at the end from awk  that
look  like errors, but they arent.   They are from a script
that builds the database for eman k.  If some optional com
ponents are not there, then some messages like the following
will be produced: cant open mana/*.*.  Ignore these.

     To quickly check the log file do:
% egrep rror log | egrep v ignored

     There should be no errors that are not ignored.
% grep rror log

     There will be several screens of  error  messages,  but
all should be followed by the phrase (ignored).

     Due  to  a some very long lines in the make output, the
log file might not be acceptable input to a text editor such
as vi.

     The target directory (/usr/esps by default) should look
like this: (this example assumes the default name)
% cd /usr/esps
% ls F













Sun;w+SRC;waves.src
tab(;); l l  l  l.   bin/;demo/;doc/;general@  lib/;man/;in


     The  bin  directory  should contain about 110 programs.
The doc directory will contain about  5  documents  (all  of
which  you  have  in hardcopy form).  The lib directory will
contain  about  30  files,  including  the   ESPS   library,
libespsg.a,  the ESPS lint library, some plot programs (that
are called by other programs), default parameter files,  and
an  example  makefile, makefile.ex.  The symbolic link, gen
eral, is a link back to the general directory in the  source
directory.

     There  is  also  a demo directory.  This directory con
tains several demo scripts.  There is a README file in  that
directory.   We  strongly  suggest that you take the time to
try these examples.  Not only do they provide  an  introduc
tion to some ESPS commands, but they serve as a check on the
installation.

     We  also  suggest  that  you use eman k (followed by a
keyword to search for) to familiarize yourself with the sys
tem.  See the eman manual page for details.

10.  Setting Up the ESPS User Environment

     If you did not install ESPS into  /usr/esps,  it  is  a
good idea to have the super user install a link from its ac
tual  location  to /usr/esps.  For example, if you installed
ESPS in your directory espsB, you could have a superuser do
the  following:   (lets  assume  your  home   directory   is
/u/smith)
# cd /usr
# ln s /u/smith/espsB esps

     After  this  is  done, all users on the system can find
ESPS files under /usr/esps.  This has the advantage, that  a
new version of ESPS can be installed and tested without dis
turbing other users on the system.  When the ESPS maintainer
is  ready  to  switch  his users over to the new version, he
simply redefines the  /usr/esps  symbolic  link.    See  the
ln(1) man page for more information on symbolic links.

10.1.  Search Paths

     In  order to execute ESPS programs users must place the
ESPS binary directory on their  default  search  path.    If
ESPS  is  installed  as  described above, that path would be
/usr/esps/bin.  Make sure that all of your  potential  users
have this on their path.















10.2.  ESPS Compilation Environment

     The ESPS install script creates several programs to aid
users  who  write their own programs using the ESPS library.
ecc invokes the C compiler with the correct options to cause
the ESPS include files and library to be searched.   It  can
be  used  in  cases where you would normally use the cc com
mand.  Additional cc options  can  be  specified  with  ecc.
The  program emake calls the Unix make command with the cor
rect environment defined.  For more information on this, see
reference [1] and the emake manual page.  A prototype  make
file  suitable  for use with emake is in the ESPS lib direc
tory.  For Masscomp systems, be sure you are in the UCB uni
verse when compiling ESPS programs.

11.  Installing Individual Programs

     The install script, by default, changes  to  each  ESPS
source directory and runs emake.  If a list of source direc
tories  are  given  on  the  install command line, then only
these will be made.   The names given, must be  exactly  the
directory  name  under  esps/general/src.  Note that this is
not always exactly the same as the  installed  name  of  the
program.

     You  can also compile (or recompile if youve made some
change) a particular ESPS program by changing to its  source
directory  and running emake install.  For example to recom
pile and install fft you would do:
% cd /usr/esps/general/src/fft
% emake install

     This will result in the makefile in that directory  be
ing  run  using the environment saved in the emake script by
the main installation script.   For more about  using  emake
and compiling ESPS programs see reference [1].

12.  Machine Dependent Programs

     Almost all of the programs in this distribution work on
all  supported  machines.   There  are, however, a few cases
where completely different versions of programs are required
on different machine types.  The  correct  program  will  be
compiled and installed depending on the value of the machine
type variable MACH.   An example of such a program is range.

13.  A Word about the Speech I/O Programs

     The  speech  input  and  output  programs,  mcplay  and
mcrecord use the C defines ADDEV, DADEV, and CLKDEV  to  de
termine  the analogtodigital device, the digitaltoanalog
device, and the data acquisition clock.  These  symbols  are
defined in the install script.  These programs are linked to
