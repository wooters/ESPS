3.2.  Complex Data Support Added to FEA Files

In  previous ESPS releases, the data types of fields in fea
ture file (FEA) records were restricted to  the  fundamental
types BYTE, SHORT, LONG, FLOAT, and DOUBLE.  In Version 4.0,
support  has  been  added  for  corresponding  complex types
BYTE_CPLX,  SHORT_CPLX,  LONG_CPLX,  FLOAT_CPLX,  and   DOU
BLE_CPLX.   Each  complex  type  has a real and an imaginary
part of the corresponding integer or floating type.  For de
tails see the FEA man page FEA (5ESPS).

At the programming level,  utility  library  functions  like
add_fea_fld,  allo_fea_rec,  get_fea_ptr,  put_fea_rec,  and
get_fea_rec have been extended so that  the  new  types  are
handled in exactly the same manner as the old ones.

All  userlevel  programs  that  process FEA files have been
modified so that, when presented with complex data, they ei
ther deal correctly  with  it  (examples  include  psps  and
addfeahd),  or  they exit with a message indicating that the
current version does not support complex data (examples  in
clude  vqdes  and distort).  Many of the userlevel programs
that currently do not support complex data  will  do  so  in
later versions.  Section 3.4 contains a list of the FEA pro
grams that currently operate in general on complex data.

done in a backwardcompatible manor  i.e., the current pro

3.4.  MachineIndependent I/O

In  previous  versions,  ESPS  files  written on one machine
could be read by ESPS programs running  on  another  machine
only  if the two machines happened to have identical numeri
cal data formats and byte order.  This meant,  for  example,
that  ESPS  files  from a Sun 4 could be read by a Sun 3 but
not by a Sun 386i or a DECStation 3100.   A  goal  for  ESPS
Version  4.0  was  that  differences  in numeric formats and
byteorder among machines on the network be hidden from ESPS
user programs.











ESPS Version 4.0c Release Notes                       page 5


To meet the goal of machine independence, we defined an  ex
ternal  ESPS  format  that  permits ESPS programs running on
different machine architectures to work cooperatively on the
same files, and we revised the ESPS file system  accordingly
to  provide  general  support  for  machine  independent I/O
(MIIO).

3.4.1.  NATIVE vs. EDR Format

ESPS files in Version 4.0 can be in one of two  format:  NA
TIVE  or EDR.  Data in NATIVE format files are written using
the native numeric formats for a particular  machine.   Data
in  EDR  (ESPS Data Representation) format files are written
in a standard representation: floats and doubles are  stored
using IEEE standard 32 bit and 64 bit floating point.  Inte
gers  and long integers are stored respectively as 16 and 32
bit twoscomplement integers.  The byte type is stored as an
8 bit byte.  Character data is  stored  in  ASCII  with  one
character  per byte.  The byte order is that of the Motorola
680X0 processors.  The ESPS external format  essentially  is
the  same  as the Sun Microsystems external data representa
tion (XDR) format.

3.4.2.  I/O of ESPS Files

ESPS programs will read files in either NATIVE or EDR format
(and also files from older versions of ESPS).   By  default,
ESPS  programs  write files in the NATIVE format for the ma
chine on which the programs  are  run.   EDR  format  output
files  can be obtained in general for all programs by creat
ing the UNIX environment variable ESPS_EDR and giving it the
value "on".  The conversion program addfeahd  (add  an  ESPS
header to an existing headerless, binary file) has an option
to specify whether the data are in NATIVE or EDR format.  At
the  programming  level,  the output format can also be con
trolled by setting the edr flag in the  ESPS  header  before
calling  write_header  (3ESPS).   (Note  that the header is
also an argument passed to the data output routines.)

The ESPS input routines can  not  in  general  handle  files
written  in native mode by other architectures  files to be
exchanged should be written in EDR format unless the  native
modes happen to match.  This lack of generality is a compro
mise  that  supports file sharing but only requires a single
machinedependent conversion module  for  each  architecture
(the  module  converts between native mode and ESPS external
format).  Some exceptions to the foregoing have been allowed
for.  In particular, the ESPS input routines will  recognize
cases  where  NATIVE  modes differ only with respect to byte
order.  Thus, for example, NATIVE format files from  a  DEC
Station  3100  or Sun 386i can be read by ESPS programs run
ning on a Sun 4 or Concurrent (and vice versa).

Using the ESPS external format can result in increased  file
I/O times, but the feature can be turned off (so that native
formats  are  always  used)  when  it isnt needed.  As men
tioned, the default is to use native formats.

3.4.3.  Typical Usage of ESPS_EDR

If you have a homogeneous network (all machines of the  same
type),  then  it is usually appropriate to have all programs
write files in NAIVE format.  Since  this  is  the  default,
nothing  special has to be done.  For heterogeneous networks
(mixed machine types), it is currently  the  case  that  the
only difference between NATIVE and EDR format is in byte or
der  (this  will change as ESPS is ported to more machines).
Since the input routines will read NATIVE format files  from
a  foreign machine in cases where the only difference in the
local NATIVE format is byte order, it  follows  that  NATIVE
format  ESPS  files from any machine on a heterogeneous net
work can be read by ESPS programs on any other machine.   Of
course,  EDR  format files from any machine can also be read
by ESPS programs on any other machine.  Thus, the choice  of
whether  to  write  ESPS  files in NATIVE or EDR format on a
particular machine should be based on efficiency,  which  in
turn is determined by access patterns.

For  example, if a set of files are produced originally on a
DECStation 3100 but used primarly by programs running  on  a
Sun  4,  it  probably  makes sense to write the files in EDR
format.  Since the EDR format happens also to be NATIVE for
mat for the Sun 4, the overhead of  byte  swapping  will  be
avoided each time the files are read on the Sun 4.

3.12.  Floating License Servers

With ESPS Version 4.0 and waves+  Version  2.0  righttouse
licenses  can be "checkedout" temporarily for use as needed
by individuals on a local area network.  Many customers have
asked for a license policy that would permit  them  to  pur
chase  licenses based on expected usage patterns rather than
based on the number of machines that happen to be  on  their
network.  The socalled floatinglicense technology provides
this flexibility.

Our  previous  license policy restricted the use of licensed
software to prespecified CPUs or servers, making it  diffi
cult for a substantial number of occasional users to share a
few  licenses.   With  floating  licenses,  licenses  are no
longer restricted to specific  hardware,  but  are  assigned










ESPS Version 4.0c Release Notes                       page 9


dynamically as needed by users throughout the network.

A righttouse license is required for each node in simulta
neous  use on a network.  (A network node is any computeen
gine with a unique host name and network  address.)   Right
touse  licenses  can  be  assigned  automatically ("checked
out") by a software license  manager  until  the  supply  of
available  licenses is exhausted.  Alternatively, if dynamic
assignment isnt appropriate, the available licenses can  be
preallocated.   An  ESPS  license checked out (or preallo
cated) to a particular node may be shared by any  number  of
simultaneous ESPS users.

The  programs echeckout (1ESPS) and efree (1ESPS) are used
to checkout and return an ESPS license.   If  a  license  is
checkout  out  on any given CPU, any user can run any number
of ESPS programs on that CPU.

The new floatinglicense technology requires that a  license
server  (elmd) daemon be run somewhere on the network.  More
information about this (and about the administration of  li
censes) is contained in the installation instructions.

5.  ESPS LIBRARY ADDITIONS AND CHANGES

5.1.   Changes  and Additions to Support MachineIndependent
I/O (MIIO)

Two items were added to the universal portion  of  the  ESPS
header  structure:  edr  and machine_code.  The former indi
cates whether the file is stored in EDR  or  NATIVE  format.
The  latter  indicates  the  machine  type that produced the
file.  See ESPS (5ESPS).

The information contained in the edr and machine_code header
items actually is stored in a short preamble that  comprises
the  first  few bytes of an ESPS file.  The read_header rou
tine first reads the preamble (which is always stored in EDR
format), determines whether the file is an ESPS file, deter
mines the file format (EDR or NATIVE), and then either exits
(if not an ESPS file) or reads the rest of  the  information










ESPS Version 4.0c Release Notes                      page 16


for  the  ESPS  header structure.  Users should keep in mind
that the C structure shown in the  man  page  ESPS  (5ESPS)
(and  defined in header.h) does not correspond to the actual
format stored in the file, which is why headers I/O must  be
done with read_header and write_header.

When  a  new  header  structure is created (by new_header or
copy_header), the edr flag in the header is set  to  1  (EDR
format) if the user has a UNIX environment variable ESPS_EDR
set to "on"; otherwise, edr is set to 0 (NATIVE mode).  This
initial  setting  can  be changed by programs (e.g., in some
cases one might want to have a commandline option to  force
a  particular format regardless of the environment variable)
before calling the I/O routines.  The format used in writing
the header is determined by the value of the edr  flag  when
the header is passed to write_header.  Similarly, the format
of  file  records is determined by the value of the edr flag
when the header is  passed  to  (for  example)  put_fea_rec.
(Yes,  you  can  get into serious trouble if you fool around
with inconsistent header structures.)

All of the library routines that support header  and  record
I/O  were  modified  to support MIIO, including read_header,
write_header, get_fea_rec, and put_fea_rec..   The  get  and
put  routines  for  the various nonFEA file types were also
modified (i.e., for SD, FILT, SCBK, and  SPEC  files).   The
I/O  routines  for  the various FEA subtypes (e.g., FEA_ANA,
FEA_SPEC, etc.) did not have to be modified since they  work
by calling get_fea_rec and put_fea_rec.

9.  COMPUTING ARBITRARY EXPRESSIONS FROM FIELDS IN  TWO  FEA
FILES

The  program  select (1ESPS) offers powerful facilities for
computing expressions on fields within FEA records.  We  are
often  asked about computing expressions that involve fields
from more than one FEA file.  In  many  cases,  the  desired
capability  can  be  achieved by combining mergefea (1ESPS)
with select.  The basic procedure  is  to  use  mergefea  to
produce  a  single  FEA  file  with fields from a variety of
source files and then to run select on  the  combined  file.
Heres   a  small  example:  Suppose  fea1  is  a  FEA  file










ESPS Version 4.0c Release Notes                      page 23


containing (among other things) a vector fld1 and  a  scalar
field  fld2.  And suppose fea2 contains a scalar field fld3.
Then the following commands will compute the  mean  of  fld1
and  add it to the log of the square root of fld2 divided by
the product of fld3 times the first element of fld1:

  %mergefea fea1 fea2 fea3
  %select e "mean(fld1) + log(sqrt(fld2/(fld3*fld1[0])))" fea3 > output

The results are stored (in ASCII) in the file output.

       programs  to  do  the X Window drawing.)  There is an
       alternate binary, /usr/esps/bin/xmcd.old, that should
       be renamed to xmcd if the standard xmcd fails.

 (3)   On Masscomp/Concurrent systems it may be necessary to
       set the DISPLAY  environment  variable  to  the  full
       hostname  of  your  system,  rather  than the generic
       unix:0.0.   For example, if the network  hostname  of
       your system is fred, then you may have to:
       % setenv DISPLAY fred:0.0

 (4)   On  systems  not  running a network, in order the for
       the license server software to operate  you  need  to
       set  the  ELM_HOST  variable to localhost, instead of
       the network hostname.   Also in this  case,  you  may
       need   to  set  the  X  Window  DISPLAY  variable  to
       localhost also (e.g. localhost:0.0).
